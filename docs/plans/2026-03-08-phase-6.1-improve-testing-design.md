# Phase 6.1 — Improve Testing: Design Document

## Problem Statement

Before adding new features (auth, multi-tenancy, WebSockets), the test and build infrastructure needs hardening:

1. **No dev/prod separation** — production Docker images contain test dependencies, test files, and dev tooling. No volume mounts for fast iteration.
2. **Verbose, non-parametrized tests** — repetitive test methods that could be collapsed with parametrize/each.
3. **Bash E2E tests** — fragile shell script with hardcoded sleeps, no visual testing, no reporting.
4. **Frontend chart color bug** — "Agent Type Distribution" donut chart renders all segments in the same color due to a fragile hex-to-name indirection in the color mapping.

## Design

### 1. Multi-stage Dockerfiles with `target` Selection

**Approach:** Single Dockerfile per service with `base`, `dev`, and `prod` stages. `docker-compose.yml` defaults to `target: prod`. A `docker-compose.override.yml` (auto-merged in dev) overrides to `target: dev` and adds volume mounts.

**Services requiring changes:**

| Service | Current | Change |
|---------|---------|--------|
| analytics-api | Single-stage, tests + pytest baked in | Split requirements.txt, add 3 stages |
| aggregation-worker | Single-stage, tests baked in | Split requirements.txt, add 3 stages |
| simulator | Single-stage, devDeps installed | Add 3 stages, dev has full deps |
| frontend | Already multi-stage (good) | No change needed |
| ingestion | Already multi-stage Rust (good) | No change needed |

**Pattern (Python services):**
```dockerfile
FROM python:3.12-slim AS base
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

FROM base AS dev
COPY requirements-dev.txt .
RUN pip install --no-cache-dir -r requirements-dev.txt
COPY . .

FROM base AS prod
COPY app/ ./app/
```

**Pattern (Simulator — Node):**
```dockerfile
FROM node:22-alpine AS base
WORKDIR /app
COPY package.json package-lock.json* ./
RUN npm ci --omit=dev || npm install --omit=dev

FROM base AS dev
RUN npm install  # full deps including vitest
COPY . .

FROM base AS prod
COPY tsconfig.json ./
COPY src/ ./src/
```

**`docker-compose.override.yml`** (auto-merged in dev, skipped in CI with `-f docker-compose.yml`):
- Sets `target: dev` for analytics-api, aggregation-worker, simulator
- Adds volume mounts: `./service:/app` for each
- Dev entrypoints use `--reload` (uvicorn) or watch mode where applicable

**Requirements split:**
- `analytics-api/requirements.txt` → runtime only (fastapi, uvicorn, clickhouse-connect, asyncpg, redis, pydantic, pydantic-settings, httpx)
- `analytics-api/requirements-dev.txt` → test deps (pytest, pytest-asyncio, pytest-httpx)
- `aggregation-worker/requirements.txt` → runtime only (redis, clickhouse-connect, psycopg2-binary)
- `aggregation-worker/requirements-dev.txt` → test deps (pytest)

### 2. Parametrized Testing

**Analytics API (`test_unit.py`, `test_integration.py`):**
- `TestFilterParsing.test_valid_periods` — for-loop → `@pytest.mark.parametrize`
- `TestDateRangeCalculation` — parametrize period → expected days
- Integration: parametrize endpoints sharing the same validation shape (e.g., all metrics endpoints returning non-empty responses)

**Aggregation Worker:**
- `test_consumer.py` `TestParseEvent` — 3 event type methods → `@pytest.mark.parametrize("event_type", [...])`
- `test_aggregator.py` — parametrize where metric assertions repeat

**Ingestion (Rust):**
- Add `rstest` to `[dev-dependencies]`
- Collapse validation tests (invalid agent_type, missing required fields) into `#[rstest]` with `#[case]` variants

**Simulator (TypeScript):**
- Use `it.each` for distribution weight tests

### 3. Playwright E2E Test Suite

**Structure:**
```
tests/e2e/
  package.json           # @playwright/test dependency
  playwright.config.ts   # baseURL: http://localhost:8080, retries, timeouts
  Dockerfile             # playwright image for containerized runs
  tests/
    health.spec.ts       # E2E-01: service health checks via API
    pipeline.spec.ts     # E2E-02: POST events → poll API until data appears
    dashboard.spec.ts    # E2E-03: navigate all 4 pages, verify charts/tables render
```

**Key design decisions:**
- Uses `request` context (Playwright's built-in HTTP client) for API tests — no curl/bash
- `pipeline.spec.ts` uses polling with `expect.poll()` instead of hardcoded `sleep 15`
- `dashboard.spec.ts` navigates the real UI, verifies page titles, chart elements, table rows
- Tests run against the full Docker stack (nginx at localhost:8080)
- Dockerfile uses `mcr.microsoft.com/playwright` base image
- Added as `e2e` service in `docker-compose.override.yml` (dev only, not prod)

**The old `tests/e2e/run.sh` is deleted.**

### 4. Frontend Chart Color Bug Fix + Unit Test

**Root cause:** `Usage.tsx` uses a double-indirection to map agent types to Tremor colors:
1. `AGENT_TYPE_COLORS[agent_type]` → hex string (`"#6366f1"`)
2. Hex string looked up in inline object → Tremor color name (`"indigo"`)

This is fragile. If any lookup fails, everything falls back to `"slate"`.

**Fix:**
- Add `AGENT_TYPE_TREMOR_COLORS` to `constants.ts` — direct agent_type → Tremor color name mapping
- Simplify `Usage.tsx` DonutChart `colors` prop to use the direct mapping
- Keep `AGENT_TYPE_COLORS` (hex values) for any non-Tremor usage

**Unit test (frontend, vitest):**
- `src/__tests__/constants.test.ts` — verify all known agent types map to distinct Tremor color names
- `src/__tests__/Usage.test.tsx` — render UsagePage with mocked data, verify DonutChart receives distinct color values (not all "slate")

This is a frontend unit test concern, not an E2E test. The E2E suite verifies the dashboard loads; the unit test verifies the color mapping logic.
